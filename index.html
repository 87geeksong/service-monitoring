02

设计方案：采集什么，如何可靠上报

围绕用户行为、关键操作、性能耗时建立统一事件模型，降低业务接入成本

采集维度

📄 页面曝光

在页面加载完成后上报页面总耗时，用于识别加载较慢的页面和整体性能趋势

👆 关键业务操作

统计按钮点击或表单提交到成功的耗时，覆盖修改催收预估、强刷刷新并保存区间、信审相似查询等场景

⏱️ 性能耗时

通过 performance.now() 计算页面加载、接口响应和渲染耗时，定位影响体验的慢环节

设计原则

  • 低侵入接入 — 业务侧只标记关键动作，通用采集逻辑由 SDK 承担
  • 模块化能力 — 页面、点击、耗时等能力按需组合，避免一次性接入过重
  • 可靠上报 — sendBeacon 优先保障关闭、跳转等场景，XHR 降级兜底；监控面板剔除异常耗时，避免极端值影响分析
  • 统一事件模型 — 标准化 eventType、eventName、timestamp、duration，便于后续聚合分析

03

前端 SDK 实现链路

前端负责标记开始、计算耗时、可靠上报，后端侧作为数据接收与看板展示承接

SDK侧：标记 → 计算 → 上报

Step 1

标记事件开始

入参：eventType（1=页面加载/2=点击/3=表单）、eventName（页面名称）

动作：记录 performance.now() 作为 eventStart，存入 Redux store

Step 2

计算耗时

动作：从 store 取出 eventStart，计算 duration = performance.now() - eventStart

输出：timestamp、eventType、eventName、timeSpent

Step 3

数据上报

接口：POST /admin/monitor/timeEvent/add

策略：优先 sendBeacon，降级 XHR，尽量覆盖关闭、跳转等容易丢失数据的场景

后端侧：轻量承接

接收上报数据

提供统一接口接收前端耗时事件，承接 eventType、eventName、timestamp、timeSpent 等字段

处理分析口径

在监控面板侧剔除异常耗时数据，避免极端值影响页面、操作和接口耗时分析

看板展示结果

按事件名称和时间范围查看耗时趋势，让前端优化有可追踪的数据依据

04

数据观测与分析

基于监控看板的耗时、频次、趋势分析，先剔除异常数据，再判断优化优先级

📈

最大耗时

含义：单个事件的最大耗时，用于发现极慢页面或极慢操作

分析：先剔除异常值，再关注峰值是否持续偏高或突然突增

⏱️

平均耗时

含义：同类事件的平均耗时，反映页面或操作的整体体验

分析：对比不同 eventName，识别长期稳定偏慢的页面或业务动作

🔢

调用次数

含义：各事件的触发次数，判断页面或操作的实际影响面

分析：结合耗时数据，高频且高耗时的事件优先处理

分析思路

1. 清洗口径：先剔除明显异常耗时，避免极端数据误导判断
2. 定位瓶颈：按「平均耗时 / 最大耗时」找出 Top 事件
3. 评估影响：结合「调用次数」判断影响面，高频高耗时优先处理
4. 验证效果：观察优化前后趋势，确认耗时是否稳定下降

05

前端性能排查工具链

监控看板负责发现问题，本地工具负责定位原因，自动化脚本负责验证回归

🔬

Chrome DevTools：定位瓶颈

  • Performance — 录制火焰图，拆分 JS 执行、渲染、布局耗时
  • Network — 查看接口和资源瀑布流，定位慢请求、阻塞和大资源
  • Coverage — 识别未使用 JS/CSS，为拆包和懒加载提供依据
📊

Lighthouse：专项体检

  • FCP / LCP — 判断首屏内容和核心内容呈现速度
  • TBT — 评估长任务和 JS 阻塞对交互的影响
  • CLS — 检查页面布局抖动，提升视觉稳定性
🤖

AI 辅助排查：提高效率

  • 辅助阅读 Performance、Network、Console 结果，快速归纳异常点
  • 把监控看板中的慢事件和本地复现场景串起来，缩短排查路径
  • 生成优化建议后仍以代码和数据验证为准，避免只凭结论修改
🎭

Playwright：复现与回归

  • 模拟真实用户点击和表单提交，复现高耗时业务动作
  • 录制 trace 对比优化前后关键路径耗时
  • 沉淀核心链路脚本，防止后续改动带来性能回退
06

性能优化手段

优化不追求“一次性全做”，而是根据监控数据优先处理高频、高耗时、影响面大的链路

01首屏资源

页面总耗时长期偏高，Network 中 JS/CSS 体积占比较大时优先处理。

  • 按页面拆分业务代码
  • 清理未使用依赖和样式
  • 大组件、低频组件延后加载

02工程化分包

构建后单个文件过大，首屏下载、解析和执行耗时明显时使用。

  • 分析打包产物体积
  • 过大的 JS/CSS 做手动拆包
  • 公共依赖、业务模块、低频页面分层加载

03第三方资源隔离

第三方 iframe 携带较多脚本，影响非相关页面加载和操作流程时处理。

  • SIP 注册相关第三方 iframe 按需加载
  • 只在指定路由初始化
  • 避免全局注入额外 JS 资源

04接口链路

操作耗时主要卡在请求等待，或同一页面存在重复接口调用时优化。

  • 合并或减少重复请求
  • 慢接口增加 loading 和兜底提示
  • 列表查询避免无效刷新

05渲染交互

Performance 中主线程任务长，页面交互卡顿或表格渲染慢时拆解。

  • 长列表分页或虚拟滚动
  • 搜索、resize、防重复点击做防抖节流
  • 避免大范围 setState 触发整页重渲染

06缓存复用

同类页面频繁重复加载基础数据，二次进入仍然耗时明显时处理。

  • 字典、配置、静态资源合理缓存
  • 查询条件和表单默认值减少重复计算
  • 避免频繁读写大体积本地数据

07图片与媒体

LCP 元素是大图，或 Network 中图片下载耗时明显时优化。

  • 压缩截图和大图资源
  • 非首屏图片懒加载
  • 按展示尺寸输出合适分辨率

08布局稳定

CLS 偏高，用户看到页面元素跳动或按钮位置变化时优先处理。

  • 图片、表格、弹窗预留稳定尺寸
  • 减少加载后插入顶部内容
  • 骨架屏和 loading 不改变主体布局

09SourceMap 定位

线上压缩代码报错难读，需要快速定位具体源码位置时建设。

  • 生产环境上传 sourcemap 到监控平台
  • 报错堆栈还原到源码文件、行列号
  • 结合版本号定位具体发布包

10监控闭环

所有优化都必须能被监控数据验证，避免只凭主观体验判断。

  • 优化前记录基线数据
  • 上线后对比平均耗时、最大耗时、LCP、CLS
  • 回归明显时快速回滚或继续拆解

后续优化建设

当前已完成总耗时监控，下一步围绕拆细指标、扩展场景、形成闭环继续建设

从总耗时拆到关键阶段

阶段补充指标排查价值
资源加载 JS / CSS / iframe 体积与加载耗时 判断是否需要分包、缓存或第三方资源隔离
接口等待 请求开始、响应返回、失败重试 区分前端等待、后端慢接口和网络波动
渲染执行 主线程长任务、表格渲染、组件更新时间 定位大列表、大表单和频繁 setState 带来的卡顿
业务操作 点击 / 表单提交到成功的分段耗时 进一步拆解修改催收预估、强刷保存、相似查询等慢链路
异常口径 异常值规则、样本量、版本维度 避免极端值污染分析，支持优化前后稳定对比

Performance API 入参口径

指标来源字段说明
最大内容绘制（LCP） PerformanceObserver: largest-contentful-paint 衡量页面主体内容加载完成时间
累计布局偏移（CLS） PerformanceObserver: layout-shift 衡量页面元素跳动带来的视觉稳定性问题
首字节时间（TTFB） responseStart - requestStart 定位接口等待、网络或服务端响应慢
资源下载耗时 responseEnd - responseStart 判断 JS / CSS / iframe 等资源是否过大
DOM 可交互时间 domInteractive - startTime 判断页面解析和脚本执行是否阻塞交互
主线程长任务 PerformanceObserver: longtask.duration 定位大列表渲染、同步计算等卡顿来源

建设路线

扩大埋点覆盖

优先覆盖高频页面和关键按钮，补齐催收、授信、配置类页面的慢操作监控

优化看板口径

按 eventName、页面、版本、时间范围拆分趋势，沉淀可复用的优化前后对比模板

异常阈值提醒

针对最大耗时、平均耗时、调用次数设置基础阈值，优先发现突然变慢的核心链路

形成优化闭环

每次性能优化都记录基线、上线时间和对比结果，让监控数据反向驱动工程化治理

09

监控埋点的业务价值

监控不是为了多记录日志，而是把线上体验转成可分析、可验证、可持续优化的数据闭环

建设前：问题靠感知

  • 页面慢、操作慢主要依赖用户反馈或人工复现
  • 只有局部现象，没有统一的耗时口径和趋势视图
  • 优化动作完成后，收益难以用同一口径证明
  • 异常耗时混入看板，容易影响数据判断

建设后：数据驱动闭环

  • 页面加载、关键按钮、表单提交统一进入监控看板
  • 按页面、eventName、时间范围分析，快速定位慢链路
  • 通过 sendBeacon 与异常值剔除，提升数据可靠性
  • 优化前后同口径对比，让工程治理形成正向反馈

发现问题 → 定位瓶颈 → 执行优化 → 验证收益 → 沉淀规范

让技术能力反哺业务流程，持续挖掘体验优化与效率提升的更大价值
